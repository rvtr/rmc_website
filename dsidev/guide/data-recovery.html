Data recovery for the Nintendo DS(i)

Want to back up cartridges, search for deleted prototypes, or revive old hardware? Meet your new "handbook" for everything to do with data recovery! First a word of warning: this is much more involved than a guide. I would strongly suggest reading the entire document, and not skipping ahead. This resource has been written so that the basic ideas are explained first, then the complex recovery processes are introduced later, building upon the previous topics. Even if you aren't making a cartridge dump, read the cartridge dump content. The tools introduced there maybe be referenced in the NAND section, for example. This is a series of lessons, not a tutorial. If you don't learn the basics, it will get confusing fast.

This resource assumes you already understand the basics of hex, and a hex editor. If not, there is a great video here.

Making a cartridge dump

When we get a new cart, the very first thing we need to do is make sure it boots. If it doesn't then it could be broken, and none of the dumping info will help you.

There are many different types of cartridges. Prototype, development, factory, demo, and retail. Most cartridges will dump properly, but there will always be a few troublesome ones we have to watch out for. Before we get to dumping, let's take a look at the known bad carts.

Problematic carts

There are also some cases where there's a DSi game flashed to a DS cart. This is the most challenging type of cart to dump, and it might be better to contact me as all dump tools will likely fail.

Now that we're aware of potential issues, we can start dumping. There are multiple tools to try:

  1. GodMode9i: for DSi or 3DS, has the best chance of working
  2. GodMode9: for 3DS only, less reliable for factory carts
  3. GodMode9 (TWL dev keys) for 3DS only, can get some troublesome DSi carts
  4. DS Game Card To SD: for DSi only, sometimes (but rarely) recovers sections that GodMode9i can't.
For options 1., 2., and 3., make sure you've dumped the .nds (not .trim.nds) and .txt. If the .nds.enc and .sav are available, grab those as well. Saves can sometimes contain essential data for ROMs to work, and the text file has the important data like the CartID.

Once you've dumped the cartridge, it's time to verify it. This will check each section of the ROM, ensuring that all parts dumped correctly. To do this, we'll use a script made using code from ntool. Place your dump in the same directory as the script, and run python ds_verify.py [path/to/ROM]. Add --dev to the command if you're working with a development signed ROM, and --encrypt for ROMs that are decrypted. If all sections are good, you're done and the dump can be released. The proper way to do a release is all the way at the end of this page. If the dump verifies but fails to boot, see the Testing Your ROM section. If the dump is bad, you'll want to try again with a different tool.

The process should always be to dump the cart, verify it, and move onto the next dump tool if the previous ones failed. Even if a dump "looks good", it may have issues. Also, remember to verify for the correct signing type (dev, retail), otherwise we'll get a lot of false negatives. If every tool failed, see Dump splicing below. If the dump verifies, but won't boot or seems to be missing necessary save data, scroll down to NAND based saves.

Dump splicing

With a cartridge like the DSi development one, sometimes all your different dumps will fail to verify. Verify each unique dump and pay close attention to the results. If all HMACs failed, we're probably dealing with a bad cart (or nothing has been written to handle it) and there's nothing that can be done. However, if some HMACs pass, there might be a chance of recovery. Look at the two figures below:

The Figure 1 dump has a bad ARM9 and Secure area, but everything else passed. Now look at Figure 1.1 which has a good ARM9 and Secure area, but bad ARM9/7i. Between both bad dumps we have all the parts to create a good dump, so let's combine them!

First we need to figure out where each section is. To do this we'll look at the dump's header. 

ARM9:
    0x020-023 : offset in ROM
    0x02C-02F : size of section
ARM7:
    0x030-033 : offset in ROM
    0x03C-03F : size of section
ARM9i: (DSi only)
    0x1C0-1C3 : offset in ROM
    0x1CC-1CF : size of section
ARM7i: (DSi only)
    0x1D0-1D3 : offset in ROM
    0x1DC-1DF : size of section
For example, we'll look at the ARM9i offset in Figure 1.2. The value is 00 D0 04 00, and it's stored as little-endian (the bytes are backwards). Flipping the bytes around gives us 00 04 D0 00, or an ARM9i offset of 0x4D000. Now we'll do the same for ARM9i size (54 4B 00 00 --> 00 00 4B 54) to get a size of 0x4B54. With all that info ready...
  • From the good dump, copy 0x4B54 bytes at offset 0x4D000
  • To the bad dump, paste + overwrite the good ARM9i into offset 0x4D000
We'll repeat this for every section until everything bad has been replaced, and until the dump entirely verifies. If the edited dump still comes out bad and you know you have every part, contact me.

NAND based saves

Some carts like DS Guide have NAND based saves. Instead of the save and ROM being on two separate chips, they're both on the same chip. This can be a huge issue for us because many tools will not know how to handle NAND based saves. Saves can be required for some ROMs to function, so we need to edit the dump tools ourselves.
First dump the cartridge with GodMode9i using the "all" option, and look at the info text (example shown below). We want to make sure that the save is not detected, and that what gets dumped is a bad size and doesn't work. You'll have to use your judgement on that.

From what we know, it also appears that you can determine NAND saves based on the CartID in the associated text files. If the second character of the CartID is '8', that means its using NAND save. The CardID's second character has been underlined below: 

Title String : NGSERVER
Product Code : UGDA01
Revision     : 0
Cart ID      : 88007FEC
Platform     : DS
Save Type    : NONE
Timestamp    : 2024-09-13 22:30:41
GM9i Version : v3.5.4
Now that you know the save is improperly detected, go into the dump options again and only dump the save. After dumping only the save you will have the correct size and everything should work. Finally, let's help out other people by adding your cartridge to GodMode9i's source code (don't worry, this is easy!). See the cardNandGetSaveSize() function? Just stick your cart in there! The last entry is the DS Guide as an example. Following the same format as the other entries, add the first three bytes of the TitleID in reverse order. Example: TID = "UGDA" --> 0x55474441 --> 0x00444755 
u32 cardNandGetSaveSize(void) {
  switch(*(u32*)ndsCardHeader.gameCode & 0x00FFFFFF) {
    case 0x00425855: // 'UXB'
      return 8 << 20; // 8MByte - Jam with the Band
    case 0x00524F55: // 'UOR'
      return 16 << 20; // 16MByte - WarioWare D.I.Y.
    case 0x004B5355: // 'USK'
      return 64 << 20; // 64MByte - Face Training
    case 0x00444755: // 'UGD'
      return 128 << 20; // 128MByte - DS Guide
  }

  return 0;
}

ntrboot carts

"ntrboot" is a special program that works as a recovery mode for DSis and 3DSes. The ROMs cannot be loaded normally (they don't even show up on the home menu), and are instead booted with a special button combination when the console is turned on. While no carts with ntrboot have been found yet, they did exist in the factory for setup and recovery. If your cartridge appears blank, you should always try looking ntrboot. The dump process will be the same as other carts, but we'll instead use a special fork of gm9 to do the dumping.

If you did get a dump, or suspect that you have an ntrboot cartridge, please contact "Rairii" on the Rare Gaming Dump discord/matrix server. We need a copy to verify the dump process.

DSi NANDs

In order to look for data on the DSi, we need to make a system backup (NAND backup). NAND is where everything is stored, from firmware to games to photos. To make a backup, follow the DSi CFW guide, but only to the point when the NAND backup has been made. You can also directly use ntrboot with SafeNANDManager, if you have a compatible flashcart.

After you make the dump, you should have a folder on your SD card named something like DT304D031319F8383C (the value after "DT" will be different for your console). Inside the folder will be nand.bin. This is your NAND backup. Now, mount the NAND backup in ninfs. You should see a twl_main.img output by ninfs. This is the main data partition where programs are stored. We'll be working almost exclusively with this file.

Backing up installed programs

Recovering undeleted apps is a pretty easy thing to do. Simply mount twl_main.img in a tool like OSFMount for Windows, natively within macOS, or GNOME Disks on linux. Once you're in, copy the nand:/title/ folder, and you're done. You can skip all the way down to the section labelled Testing Your ROM.

Recovering deleted apps

Before we get into the "real" data recovery, it's important to understand how file deletion works. When you write a file, the filesystem will create a bunch of pointers. These pointers tell us information about the file and where it is on the NAND. If I were to delete the file, the pointers would get removed, but the actual data of the file would remain. Most times it's possible to fully restore files. However, if I were to write new files to NAND, the deleted data is unprotected and could easily get overwritten by new data.

When recovering data, it's very important not to touch anything. Do not install new programs. Do not region change. Do not run factory tools. Anything could destroy your deleted data forever!

So, how do we find deleted apps? Simple! Every single piece of software has a header. Like you saw with cartridge dumping, headers tell us all sorts of information about the app. While some of this data will change (sizes, offsets, names), there are always some bytes that never change. For our purposes we will be looking at the ASCII value of WfA (hex 0x576641) as shown in figure 3. Start by seaching for WfA in twl_main.img of your decrypted NAND dump.

Once you find a match for WfA, look at the surrounding bytes. WfA is always 0x60 bytes after the start of the header, so let's confirm that we've found the correct data.
  • The header should start with 0x0C ASCII bytes (or zerobytes) making up the app's short title. In figure 3 this is KART
  • After the short title, look for the Title ID. The TID is 0x04 ASCII bytes. In figure 3 this is NTRJ
  • Finally confirm the publisher code. This is 0x02 ASCII bytes. In figure 3 this is 01
If your possible header looks like figure 3 and matches the short checklist, it's a header. Now it's time to extract the software. If you are working with a TWL ROM (DSiWare) take a look at 0x210-0x214 in the header. If it's an NTR ROM (child, see below), then look at 0x80-84. The value for your ROM type is the total ROM size, written in little-endian. Like last time we'll flip the bytes around (in figure 3 this is 0x00CE0400 --> 0x0004CE00). From the start of the header we will select the number of bytes specified in the total ROM size. Save the bytes as a new file and you're done! If all goes well your app should boot.

ⓘ Tip: Beware of The Childs!

Watch out when searching for headers! Most headers you will find are for full apps, but some are "download play childs". These are embedded apps used in multiplayer. For example, Mario Kart DS allows players to send a shortened copy of the game to another person who does not own the Mario Kart DS cartridge. Sometimes the childs may embedded in recoverable apps in NAND, and other times are instead just leftovers from someone playing multiplayer games. Childs are often untestable and unbootable on their own, so only recover them if you believe that they are a part of a larger application.

You can identify a child by the header. Take a look at Figure 3.8. Do you see the extra header region shown in red? This only appears on DSi ROMs and DSiWare (TWL applications). If you do not see the extra header region, this is a normal DS (NTR) program and must be a child.

Fragmentation: Detecting Patterns

Now we know the basics of data recovery... but it's not always so easy as it seems. When files are created, they may not be continuous, and so you can't simply copy them out like before. A new file might start writing in NAND's free space, but before it finishes it will hit an existing file (often times system files on the DSi). What happens? The new file "jumps" over the old data, and continues right after. That is called fragmentation. Fragging may occur many times in a single file, and it's our job to find every piece and stick them back together.

Finding each part of a fragmented file is largely looking for patterns. Every file will have very recognizable data types. There might be long sections of random data, then repeating headers, then some text, etc. We can pay attention to what the patterns are, and if there's an abrupt change then it may mean we've hit a fragment, or we're on a new region of the file.

Let's look at Figure 3.3 for an example. Notice how the data is completely random up until 0xC14000. Afterwards, there's a repeating pattern of 4 ASCII bytes, 4 zerobytes, and 20 bytes of randomness. Since the change in datatypes is so different, it would be a good idea to mark data after 0xC14000 as a fragment. To be extra sure, let's compare the "bad" data after 0xC14000 to undeleted files in NAND. In this specific case the data comes from the installed "DS Cart Whitelist". If we take the whitelist and compare it to the file we want to recover, we can find all the data belonging to the whitelist and remove it. Everything that remains should be the data we want. I'll talk about this more later.

Searching for patterns in letters and numbers can be a bit challenging, so sometimes it's good to use a "binary visualizer". This allows us to see the file as colors. For ease of access we'll try the web visualizer called binvis. There are multiple different ways to visualize data (click the paintbrush icon):

  • Byteclass: bytes are colored based on their type. Low, high, ASCII, 0x00, and 0xFF bytes will each get their own color (no gradient used). This is the best for finding padding and text.
  • Magnitude: bytes are colored in a gradient based on the value. Higher bytes will be light blue, and low bytes will be colored black.
  • Entropy: bytes are colored based on how much bytes vary (their randomness). If there are many different values and patterns, the file is high entropy, and it will be bright magenta. If few unique values are used and there are patterns, the file is low entropy, and will be darker.
  • Detail: each value is assigned as different of a color as possible, making it easy to spot individual values.

Viewing file entropy is one of the best tools we have available because patterns stand out. Looking at Figure 3.4 you can see an entropy view of the whitelist fragmentation. The file we were trying to recover had a lot of randomness, as you can see by the bright magenta. Now look at the highlighted region. Notice how it is much darker, and it stands out in the randomness? Just based on the color, you can tell this is the whitelist data which does not belong.

There are of course a few things you need to keep in mind. Manual searches and binary visualizers find most of the fragments, but it's easy to get confused by what is a different file, and what is a new section of the current file. Figure 3.4 has many more different patterns visible. It's up to us to make the final call and use our best judgement on what "feels right" to include or remove. Thankfully there are some rules to help narrow down data:

  • Files and fragments appear to always start on an address ending in 0x200. So this could be 0x19A5200, 0x2D99200, 0x699200, etc.
  • The NAND is separated into sectors with a size of 0x200. You can think of these as little evenly placed slots the data fits into. A file or fragment can only take up space equaling to a multiple of 0x200, can never start partway through a sector, and no two files will ever share the same sector.

System Files

Before we do fragmentation recovery on any device, we need to get familiar with system files. These will most often be what causes a fragment, so recognizing system files will help remove the bad data from the recovered file.

  • Save files (Fig. 3.5): these always come in the form of FAT12 partitions. They will have a recognizable ASCII string of xV4.
  • Certificates (Fig. 3.6): each one will always be 0x300 bytes, and they're often strung together multiple at a time. They can be identified by ACSII Root-CA0000000x
  • TMDs (Fig. 3.7): these are 0x208 bytes. They can be identified by ACSII Root-CA0000000x-xx0000000x. The structure looks similar to certificates but with the added identifier connected by a dash.
There are many types of system files, but the ones below are the most common. Being aware of even just a couple formats will make the recovery process much faster. You might want to look at the nand:/sys/ and nand:/sharedX/ in your dumps for more examples.

Deleting what you know

When recovering a file with many fragments, it's going to be hard to find every part. Instead of searching and guessing, try to delete what you know. Is your file fragmented by an installed app? Identify that app. Once you know what it is, delete any trace of it by filling its contents with zerobytes. Now do the same with the next interrupting file. By zerofilling unrelated data repeatedly, you carve out all the space around your file, and eventually you will reach a point where the only data left is pieces of your file (or parts of other deleted files). Figure 3.9 shows the results of zerofilling in binvis, where the colored data is the leftover fragments.

ⓘ Tip: Trouble determining if a fragment belongs?

Typically the content of a ROM will follow a common pattern. You'll have contents like the program code (medium entropy), assets (high if compressed, low if not), and text (very low). All of these will be confined to specific regions, and similar data types will be near each other. If you try to look into what the bytes are, you should be able to pick apart each region.

Whenever you find data that doesn't match its surroundings, like in Figure 3.8, it's usually a sign of bad data. There are of course exceptions, but most times you want to look for what stands out, and delete it.

Deleting can also extend to some negative decryption. When the NAND is initially set up, it is filled entirely with dummy zerobytes. This seems fine, but the partition (and the zerobytes) are technically "encrypted". That means that those zerobytes, despite not being any meaningful data, will still be decrypted when the partition is read by tools like ninfs. Decrypting zerobytes essentially encrypts them, turning them into high entropy, or random data. We can see this in action in Figure 3.10. It's important to compare the decrypted partition to its location in the raw NAND, clearing out this garbage data.

Putting fragmentation into practice

Now that you've figured out how to locate fragments, collect every single one you can find. After collecting enough fragments, we can start to re-assemble the original file. With linear fragmentation (what you'll usually see), pick every fragment you have identified by the order they appear in. Combine each of these fragments into a single file. This will be your work ROM.

As you continue adding to your work ROM, run the verification script. For a refresher, you run the script as python ds_verify.py [path/to/ROM], with the added --dev option for development/prototype ROMs. First start with the header, then look at each region of the ROM as they appear. By going one by one, you can find mistakes before you put too much time into the recovery. For example,if the ARM9i section is bad, you can look at the offsets where ARM9i is in NAND. There would likely be a fragment of another file in the ARM9i. Now you only need to review that single section. If you had built the entire ROM and then verified, you'd have multiple bad sections and it would be harder to know where to start.

Testing Your ROM

Testing ROMs is a very unreliable process, especially with prototypes and development apps. Each program will have very specific requirements in order to boot. We won't know what those requirements are, so we need to learn and attempt all methods of testing.

melonDS

First, let's set up a testing environment with the melonDS emulator. I won't explain how to set up the emulator (this is easy enough to figure out yourself), but I will give you the best files. The most reliable results come from using the full BIOS, which I have prepared here. Only the full BIOS will support DSi NANDs with prototype bootloaders, and some prototype games. Any other BIOS you download online will not work. In order to actually make use of the full BIOS, the "Full BIOS boot" option must be enabled.

For NANDs, I suggest using my NAND generator. This will create a clean NAND from scratch, eliminating any issues that might come from a used console. Both retail and development NANDs can be created this way. You can obtain the IDs and HWInfos needed for the generator here. However, keep in mind that these NANDs are not legitimate, and come with unlaunch pre-installed.

Once melonDS has been set up, try your dumped/recovered program with each of these boot methods:
  • Attach the ROM as a cartridge, then autoboot it
  • Attach the ROM as a cartridge, load the home menu from unlaunch, then boot the cartridge
  • Attach the ROM as a cartridge, then boot it from the home menu (no unlaunch installed)
  • Copy the ROM to the virtual SD, then boot it with unlaunch
  • Copy the ROM to the virtual SD, then boot it with TWiLight Menu++
  • For DSiWare: copy the ROM to the virtual SD, then install it to NAND with NTM
  • For DSiWare: create a TAD with maketad, copy the TAD to the virtual SD, then install it to NAND with TDT. For dev NANDs, you will need the development build of TDT.

Real hardware

Real hardware is a bit more reliable, especially if you have development units and cartridges. For standalone DSis (retail or dev), try the following methods:
  • Copy the ROM to the SD, then boot it with unlaunch
  • Copy the ROM to the SD, then boot it with TWiLight Menu++
  • For DSiWare: copy the ROM to the SD, then install it to NAND with NTM
  • For DSiWare: create a TAD with maketad, copy the TAD to the SD, then install it to NAND with TDT. For dev units, you will need the development build of TDT.
If you have one, development cartridges are the best tools for testing. You can flash ROMs like developers would do with prototypes. This will help create the most accurate environment for running software. Eevitronics has created a flasher for the 3DS.

Once your dev cart is ready, you can either boot it as is from the home menu, or you can boot it from Simple DS Slot-1 Launcher.

ⓘ Tip: What if my ROM came from a development cartridge?

If your ROM came from a development cartridge, DO NOT use that same cartridge for testing. If you overwrite the cartridge with your dump before confirming that the dump was correct, you could write bad data and destroy the original software. As for testing, the point is to ensure that the dumped ROM is valid. We need to test the dump specifically, not the cartridge.

If you have a second and blank development cartridge, you may write to that and proceed with testing.

Releasing Data

Everyone has their own preferences on how to release things, and there's no perfect way to go about it. Some people do one hash, others do six hashes. Some list where the ROM came from, others don't. Below are my rules for doing a release. Following them is not required, but it is highly recommended. The rules help you, and people viewing your release.

  • Always declare if software was deleted and recovered. If there are issues found with the ROM later on, people need to know the history of the ROM
  • Also declare if software was recovered from a physically repaired cartridge
  • Share the ROM with photos of the cartridge (front and back), or the development unit it came from
  • Include hashes with the ROM. SHA-256 and MD5 should be enough
  • Submit your dumps to no-into. @hiccup3675 on Discord can help you with this, or register on the forum and post to the "New Dumps & Redumps" sub-forum.
You can make the release in any number of locations. Hidden Palace will accept some dumps, the Internet Archive is super easy to work with, and I can release your dumps on this website as well. There are also some twitter accounts which release prototypes, but I find that they're inconsistent with being active.
